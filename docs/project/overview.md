# Project Overview

Последнее обновление: 2026-03-15

## Overview

`kanatka2` — программа автоматического отбора лучших фотографий с горнолыжной канатной дороги. Продукт на продажу.

Камера снимает проезжающих лыжников. Программа группирует кадры в серии по времени, отбрасывает пустые кресла, выбирает лучшее фото в каждой серии, собирает печатные листы (2x4) и отправляет в типографию по сети.

## Architecture

Сейчас есть **реализованная схема** и **новая рабочая гипотеза**, которые нужно развести явно.

### Реализовано в коде на 2026-03-15

| Компонент | Роль | Статус |
|-----------|------|--------|
| **Основная программа** | Обработка фото, веб-интерфейс, сборка листов | Рабочий прототип |
| **Программа-приёмник** (`receiver/`) | Лёгкий просмотр листов в типографии | Реализовано, но подлежит переоценке |

### Рабочая гипотеза после нового уточнения автора проекта

| Узел | Функция |
|------|---------|
| **Верхний компьютер у камеры** | Малый компьютер в ящике у верхнего конца канатки. Получает фото от камеры и передаёт их по сети |
| **Нижний компьютер у принтера** | Компьютер в помещении канатки, где сажают/высаживают людей. Делает отбор, сборку листов, UI и печатный workflow |

Следствие: компьютеров действительно два, но основной продукт почти наверняка должен концентрироваться на нижнем компьютере. Верхний компьютер пока выглядит как capture/transfer node, а не как место основной логики отбора.

### Подтверждено ответами автора по checklist

- Основной продукт должен работать на нижнем ПК у принтера.
- Верхний ПК у камеры в основном передаёт файлы по сети и не занимается основным отбором.
- Входная папка для основной программы должна быть настраиваемой.
- Отдельный `receiver/` не нужен как обязательная часть поставки.
- nearby/rescue — штатная функция основной программы, но должна быть отключаемой.
- Самое важное исправление сейчас: `score` и выбор лучшего кадра.

### Recommended Topology

```text
[Камера]
   ->
[Верхний малый ПК у канатки]
   Роль: принять файлы от камеры и положить их в сетевую папку
   Не делает: score, выбор, листы, операторский UI, печать
   ->
[LAN / общая папка]
   ->
[Нижний ПК у принтера]
   Роль: основная программа
   Делает: мониторинг входящей папки, группировку серий, score, выбор,
           rescue/ручной разбор, сборку листов, печать, архив/очистку
```

### Product Boundary

**Обязательная часть продукта:**
- Основная программа на нижнем ПК.
- Работа с сетевой входящей папкой как источником фото.
- Локальные папки нижнего ПК для `selected`, `rejected`, `sheets`, `archive`, `logs`.
- UI для просмотра серий, спорных случаев и управления печатным потоком.

**Пока вне основного ядра:**
- Сложный отдельный UI на верхнем ПК.
- Богатая логика на верхнем ПК, кроме захвата/передачи файлов.
- Обязательный отдельный `receiver/` как второй полноценный продуктовый компонент.

### Deployment Modes

| Mode | Что ставим | Когда использовать |
|------|------------|-------------------|
| **Primary mode** | Один основной app на нижний ПК | Целевой режим по текущему пониманию |
| **Legacy/optional mode** | Основной app + `receiver/` | Только если позже подтвердится отдельный операторский экран или удалённый просмотр листов |

**Реализация UI:** нативное окно через pywebview (WebView2/Edge), HTTP-сервер на localhost:8787. Tkinter GUI сохранен как fallback.

**Поставка:** два EXE-файла (основная программа + приёмник) через PyInstaller. Это KAN-019.

Верстка: Desktop Full HD (1920x1080). Мобильная адаптация — отдельная задача.

## Lock Zone

Логика отбора (двухэтапная детекция):

| Шаг | Что проверяем | Правило | Результат |
|---|---|---|---|
| 1 | Есть ли лицо в кадре | MediaPipe face detection | Лицо найдено → score по лицу |
| 2 | Fallback: есть ли человек | Haar upper body cascade | Человек найден → score по общему качеству кадра |
| 3 | Резкость | Laplacian variance (лица или кадра) | Участвует в score (вес 35) |
| 4 | Освещение | Яркость (лица или кадра), центрирование вокруг 145 | Участвует в score (вес 25) |
| 5 | Выбор лучшего | Кадр с максимальным score | 1 кадр → selected, остальные → rejected |

Веса: `person_present=40`, `sharpness=35`, `exposure=25` (сумма=100).

Fallback: если MediaPipe не нашёл лицо (очки, шлем, ракурс), Haar upper body cascade ищет силуэт верхней части тела. Если найден — кадр считается «занятым», score рассчитывается по общему качеству изображения.

Ограничение текущей lock-zone: код уже собирает `yaw/pitch/roll`, `ear`, `mouth_ratio`, но итоговый score их не использует. Это кандидат на переработку в KAN-038.

Продуктовое направление для KAN-038:
- сначала проверка `occupied chair`;
- потом отсев явного брака;
- потом ранжирование пригодных кадров;
- `smile` — только слабый бонус или tie-break, не основной фактор.
- если на кресле несколько людей, при прочих равных выигрывает кадр, где оба/все читаются лучше;
- кадр с более читаемыми лицами должен побеждать кадр, где лица сильнее в профиль;
- горнолыжные очки, маски и шлемы считаются нормой, а не браком;
- частично перекрытое тело внизу допустимо, если верхняя часть и головы читаются.
- для KAN-038 подготовлен отдельный design brief: `docs/project/kan-038-score-design-brief.md`.
