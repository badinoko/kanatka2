## Что убрать перед production-деплоем

Следующие фичи добавлены специально для тестовой фазы (v2.0) и **должны быть убраны или отключены** перед финальным развёртыванием на объекте:

1. **Автозапуск симулятора при Start мониторинга** (`_run_inbox_simulation` в `series_browser.py`, `/api/simulate` endpoint).
   - В реальной работе фото поступают от камеры в `incoming/` напрямую — симулятор не нужен.
   - Убрать вызов `fetch('/api/simulate')` из `toggleMonitor()` в JS.
   - Убрать `_SimulatorState`, `_run_inbox_simulation()`, `/api/simulate` endpoint.
2. **Кнопка «▶ Запустить» без симулятора** — после удаления симулятора кнопка просто запускает мониторинг, как и было до v2.
3. **`INBOX/` в инсталляторе** — папка для тестовых фото заказчика. При деплое на объекте можно убрать из поставки или оставить как резервный вариант.
4. **`simulate_camera.bat`** — не нужен в production. Убрать из `photoselector.iss`.

## Подтверждённая рабочая модель

Физически в системе два компьютера:
- верхний узел у камеры;
- нижний узел у принтера.

Текущее рабочее понимание после ответов автора и разговора с заказчиком:
- верхний ПК в основном получает фото от камеры и передаёт их по сети;
- нижний ПК делает основной workflow;
- оператору не нужен поток фото в реальном времени;
- основной выход системы — автоматически напечатанный лист;
- по листам сразу запускается производство сувениров;
- серия должна формироваться по времени создания файла;
- все файлы одной серии должны укладываться в окно не больше 2 секунд;
- `nearby/rescue` нужен как инструмент исключений, когда автомат ошибся или клиент просит вариант получше.

Следствие:
- система не должна ждать человека по умолчанию;
- человек вмешивается редко и точечно;
- качество автоматического выбора кадра становится главным узким местом продукта.

## Архитектурная трактовка

### Основной режим

```text
[Камера]
  ->
[Верхний ПК]
  Роль: принять файлы и передать их по сети
  ->
[Общая папка / сеть]
  ->
[Нижний ПК]
  Роль: мониторинг, grouping, scoring, selection,
        sheets, print, rescue, archive, operator UI
```

### Что считать ядром продукта

Обязательная часть:
- основная программа на нижнем ПК;
- настройка сетевой входящей папки;
- автоматический отбор лучшего кадра;
- сборка листов;
- автопечать;
- ручной rescue по исключению;
- инженерный режим настройки и проверки.

Пока optional:
- отдельный `receiver/`;
- отдельный полноценный UI на верхнем ПК;
- отдельный viewer на другом рабочем месте.

## Что означает это для UI

Главное уточнение: полезный GUI оператору всё ещё нужен, просто не в роли live-photo wall.

Неправильная модель:
- постоянно гнать поток фото на экран оператора;
- заставлять оператора следить за каждой серией;
- делать человека обязательным шагом перед печатью.

Правильная модель:
- экран работает как control panel и exception console;
- основной поток идёт автоматически;
- интерфейс помогает понять, что система жива, что печатается и где нужно вмешательство.

### Полезный GUI на нижнем ПК

Реально полезные блоки:
- статус мониторинга входящей папки;
- статус очереди, печати и последнего сформированного листа;
- список спорных или rescued серий;
- быстрый вход в страницу серии (rescue через «Заменить»);
- настройки инженера;
- тестовый режим без реальной печати.

### Нагрузит ли это машину

По текущей логике продукта небольшие превью и обычный GUI не являются главным источником нагрузки.

Основная нагрузка идёт от:
- OpenCV / MediaPipe;
- чтения и записи файлов;
- сборки листов;
- реальной печати и сетевого I/O.

Значит:
- лёгкий интерфейс с превью, статусами и последним листом допустим;
- опасно не само окно, а попытка превратить его в постоянный live-stream всех кадров.

## Вмешательство оператора и rescue

Функционал «Рядом» (`/nearby/`) удалён (KAN-081) — выбор фото из другой серии избыточен.

Остаётся механизм «Заменить» (KAN-082): кнопка на странице конкретной серии.

Три сценария вмешательства оператора (все используют одну кнопку «Заменить»):
1. Оператор видит неудачный автовыбор → нажимает «Заменить» на другом фото серии.
2. Система сама помечает серию как ambiguous → оператор подтверждает или выбирает другое.
3. Клиент на стойке просит другое фото → оператор находит серию → lightbox → клиент выбирает → «Заменить».

Во всех случаях «Заменить» запускает bump-механику (KAN-089):
- выбранное фото вытесняет одно фото из текущего незапечатанного листа;
- лист немедленно уходит на печать;
- очередь сдвигается на 1 фото;
- мониторинг не прерывается.

Практический вывод:
- нужен статус `ambiguous_manual_review`;
- нужна очередь исключений;
- основной поток не должен блокироваться.
- `delta_score`, вызывающий помощь оператора, должен настраиваться в settings.
- спорные серии должны полностью выходить из основного workflow до решения по ним.
- спорные серии не должны задерживать формирование остальных листов.

### Что подтвердил автор в raw decision-note

По временной версии документа с комментариями автора подтверждено буквально:
- вариант A по operator queue исключается;
- вариант B остаётся рабочим;
- вариант C по queue допустим только как более позднее расширение;
- cleanup variant C подтверждён напрямую.

Это значит, что по этим трём вопросам мы уже почти не гадаем, а работаем от подтверждённой рамки.

## Score и настройка на объекте

Текущая проблема не в том, что настроек много, а в том, что score слишком хрупкий и слишком чувствителен к порогам.

Наблюдение пользователя по датасету подтверждает это:
- сдвиг порога на `0.05` уже заметно меняет выдачу;
- иногда пролезает пустое кресло;
- иногда хороший кадр проигрывает соседнему.

Из этого следуют две задачи:

### 1. Переписать саму score-модель

Нужна модель, где:
- пустое кресло не может выиграть у нормального кадра;
- fallback не доминирует над face-candidates;
- более читаемые лица важнее косметических бонусов;
- несколько читаемых людей на кресле учитываются лучше;
- неуверенный случай помечается как `ambiguous`, а не притворяется уверенным выбором.

Для этого уже есть отдельный технический документ:
- `docs/project/kan-038-score-design-brief.md`

### 2. Сохранить инженерный tuning

Не все текущие настройки обязательно должны пережить переработку, но сама идея полевой подстройки правильная.

Нужно оставить:
- параметры группировки серий;
- ключевые detection thresholds;
- включение/выключение rescue;
- включение/выключение автопечати;
- безопасный тестовый режим.

Не нужно оставлять:
- лишние дублирующие параметры без ясного эффекта;
- настройки, которые инженер не сможет осмысленно калибровать на объекте.

## Test Mode и экранная эмуляция печати

Это хорошая идея и её уже можно считать нормальным направлением продукта.

Что даёт test mode:
- позволяет настраивать thresholds без реального расхода бумаги и материалов;
- показывает, что именно ушло бы на лист;
- позволяет быстро увидеть пустые кресла, промахи и последствия изменения порогов;
- даёт живой, полезный GUI даже в инженерном режиме.

Минимально полезный вариант:
- лист формируется как обычно;
- вместо отправки на принтер открывается экранный preview;
- preview можно увеличить;
- поверх листа можно включать debug overlays и scores;
- боевой выход при этом не трогается.

Это отдельная задача KAN-051.
