## 2. Что важно прямо сейчас

- В проекте есть архитектурная развилка.
- Исторически код и docs были настроены под схему:
  - обработка на верхнем компьютере;
  - отдельный приёмник листов на нижнем компьютере.
- Новая вводная от автора проекта на 2026-03-15:
  - мощный нижний компьютер у принтера должен делать всю обработку;
  - верхний компьютер в ящике только складывает фото в сетевую папку;
  - сервис верхнего компьютера пока вне скоупа.

Рабочая гипотеза для следующих сессий:
- основной продукт = одна главная программа на нижнем компьютере;
- `input_folder` может быть сетевой папкой от верхнего компьютера;
- `receiver/` пока не удалять, но считать legacy/optional режимом, пока пользователь явно не подтвердит обратное.

## 3. Как подходить к задачам

1. Сначала зафиксировать решение в `docs/project/overview.md`, если меняется архитектура, приоритеты или продуктовая форма.
2. Длинные объяснения и этапы держать в `docs/project/roadmap.md`, а не раздувать `overview.md`.
3. Потом append-only запись в `docs/project/progress.md`.
4. Если меняется точка входа в следующую сессию, обновить `docs/project/startup.md`.
5. Только после этого делать кодовые правки, если направление уже определено.

## 4. Где сейчас главный риск

- Не в упаковке и не в UI.
- Главный риск в том, что текущий score упрощён до 3 компонентов, хотя код уже собирает более богатые face-метрики:
  - `yaw/pitch/roll`
  - `ear`
  - `mouth_ratio`
  - локальная резкость и яркость лица
- Из-за этого fallback-кадр без найденного лица может выигрывать у кадра с лицом просто по общей резкости и яркости.

Если задача связана с «почему выбралась не та фотография», первым делом использовать `skills/kanatka-score-debug`.

## 5. Практический порядок работы

Для архитектурных задач:
- читать `CLAUDE.md`, `overview.md`, `startup.md`, tail `progress.md`;
- затем проверять фактическое состояние файлов, а не верить старым статусам в docs;
- не удалять `receiver/`, пока пользователь явно не утвердил новую схему.

Для задач по качеству отбора:
- смотреть `workdir/logs/ser*_report.json`;
- сравнивать `selected` с `rejected`;
- отделять проблему threshold tuning от problem scoring logic.

Для end-to-end проверки:
- использовать `skills/kanatka-pipeline-smoke`.
