# CODEX.md

Основной рабочий регламент Codex для репозитория `kanatka`.

Назначение: короткий и живой справочник по тому, как вести этот проект, где хранить состояние работ и как не терять контекст между сессиями.
Последнее обновление: 2026-03-14.

---

## 1. Приоритет правил

1. `AGENTS.md`
2. Этот файл (`CODEX.md`)
3. Живая проектная документация:
   - `docs/project/overview.md`
   - `docs/project/startup.md`
   - `docs/project/progress.md`
4. `TZ_PhotoSelector.md`
5. Код и конфиги репозитория

Важно:
- `CLAUDE.md` больше не является рабочим документом проекта.
- Состояние проекта должно сохраняться в `docs/project/*`, а не только в истории чата.

## 2. Базовые правила работы

1. Не объявлять задачи завершенными автоматически.
   - После собственных изменений использовать рабочую формулировку: изменения внесены, требуется проверка пользователя.

2. `overview.md` является главным дашбордом проекта.
   - Новые заметные направления работы, решения и хвосты сначала должны появляться там.

3. `progress.md` всегда append-only.
   - Не переписывать старые записи.
   - Каждая существенная сессия должна оставлять короткий след: что изменено, что осталось проверить, что делать дальше.
   - Если журнал дорастает примерно до 1000 строк, не чистить его вручную и не урезать на месте.
   - В этом случае целиком переносить старый `progress.md` в `archive/`, а на его месте создавать:
     - краткий `summary` со ссылкой на архив;
     - новый облегченный `progress.md` с перекрестной ссылкой на архивную копию.

4. `startup.md` нужен как handoff-файл.
   - Он должен помогать быстро войти в следующую сессию без чтения всего репозитория.

5. Не удалять задачи из `overview.md` без явной команды пользователя.

6. Если работа уходит в сторону от текущего дашборда, сначала обновить документы, потом расширять код.

7. Пользователь не обязан отвечать на технические вопросы.
   - Там, где возможно, принимать технические решения самостоятельно и фиксировать их в документах.

8. Ничего не удалять из проекта.
   - Любые документы, результаты чисток, старые артефакты, мусорные файлы и устаревшие выводы переносить в `archive/`.
   - Если структура архива потребует разбиения по подпапкам, делать это внутри `archive/`, а не через удаление данных.

## 3. Текущий рабочий контекст проекта

Проект: локальная программа отбора лучших фото с горнолыжной канатной дороги.

Факты, на которые опираемся сейчас:
- стек: Python 3.11+, OpenCV, Pillow, MediaPipe, tkinter;
- решение CPU-only, под Windows;
- есть рабочий прототип пайплайна: анализ кадров, скоринг, выбор лучшего, сохранение результатов, сборка листов;
- недавно добавлены:
  - series-aware fallback для кадров, где в серии лица потерялись на части снимков;
  - отладочная таблица score поверх фото вместо компактного бейджа;
  - чекбокс GUI для включения и выключения отладочного оверлея.

Открытые вопросы текущего этапа:
- часть серий с людьми может уходить в `empty`, если MediaPipe не находит лица ни на одном кадре серии;
- per-photo JSON уже отключены по умолчанию, но схема общих логов и путей еще требует дальнейшей стабилизации;
- конфигурация путей и логов пока вынесена в `config.json`, но требует более удобной настройки и документирования.
- базовый алгоритм отсевa и оценки теперь нужно держать синхронным с `overview.md -> Lock Zone`, чтобы не переопределять его заново в каждом чате.

## 5. Правила обновления документов

1. После существенных кодовых или организационных изменений обновлять минимум:
   - `docs/project/overview.md`
   - `docs/project/progress.md`

2. Если изменился способ продолжения работы или приоритеты, обновлять и `docs/project/startup.md`.

3. Если пользователь просит “навести порядок” или “зафиксировать состояние”, сначала обновлять docs bundle, затем уже переходить к следующему функционалу.

4. Если пользователь подтверждает результат явно, это можно отразить в `overview.md`.
   - Без подтверждения не использовать формулировки окончательной завершенности.

5. Любое изменение базовой логики отбора сначала отражать в `docs/project/overview.md` в lock-zone, а уже потом в коде.

## 7. Практические правила разработки

1. Сначала искать минимальное решение, потом расширять архитектуру.
2. Все изменения по скорингу сопровождать проверкой на реальных фото или на журнале `workdir/logs/ser*_report.json`.
3. Для объяснения оценок использовать:
   - исходное фото;
   - `.json` рядом с фото;
   - `workdir/logs/ser*_report.json`;
   - `workdir/logs/annotated/...`.
4. Новые временные debug-механизмы допустимы, если они документированы и потом могут быть отключены.
5. Крупные решения сначала кратко фиксировать в `progress.md`, чтобы не терять контекст из-за длины сессии.

## 8. Что не делать

- Не возвращать `CLAUDE.md` как основной источник правил.
- Не оставлять критичные промежуточные решения только в чате.
- Не объявлять “все готово” без проверки пользователя.
- Не удалять существующие артефакты, логи и документы.
- Не сокращать `progress.md` вручную без переноса полной копии в `archive/`.
